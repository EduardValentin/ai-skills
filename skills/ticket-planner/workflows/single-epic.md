# Single Epic Workflow

Use when the user wants one epic for a feature area.

## Steps

1. Identify the feature area from the user request and available source context.
2. Read [../references/ticket-quality.md](../references/ticket-quality.md) and, when a prototype exists, [../references/prototype-boundaries.md](../references/prototype-boundaries.md).
3. Deep-dive the relevant PRD sections, prototype routes/states, feature brief, plan, or approved interview brief.
4. Ask only blocking clarifying questions, batched by topic.
5. Draft the epic using the epic template:
   - Title
   - Overview
   - Scope
   - Candidate story titles
   - Source context references
   - Assumptions or open questions
6. Refine until the user explicitly approves the exact epic draft.
7. Ask whether to plan vertical-slice stories for the epic.
8. If the user wants to publish the approved epic, infer the issue tracker from project docs or ask, then switch to the relevant tracker-specific publishing skill.

## Quality Bar

The epic should define a coherent product capability, not a UI area. If the epic is just "build this screen," reframe it around the user outcome and use the prototype only as the design reference.
